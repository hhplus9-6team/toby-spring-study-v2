# 1.1. IoC와컨테이너: 빈 팩토리와 애플리케이션 컨텍스트

## 1. IoC 컨테이너 : ApplicationContext, BeanFactory
스프링 애플리케이션에서는 오브젝트의 생성과 관계설정, 사용, 제거 등의 작업을 애플리케이션 코드 대신 독립된 컨테이너가 담당한다. 코드 대신 오브젝트에 대한 제어권을 갖고 있다고 해서 IoC라고 부른다.

스프링에서 IoC를 담당하는 컨테이너는 ApplicationContext 또는 BeanFactory 라고 블리운다.


## 2. BeanFactory

`BeanFactory`는 스프링 IoC 컨테이너의 가장 기본적인 인터페이스이다.

이름 그대로 빈을 생성하고 관리하는 팩토리 역할을 한다.

```java
BeanFactory beanFactory = ...;
UserService userService = beanFactory.getBean("userService", UserService.class);
```

`BeanFactory`가 제공하는 핵심 기능은 다음과 같다.

* 빈 생성
* 빈 조회
* 의존관계 설정
* 빈 생명주기 관리

`BeanFactory`는 스프링 IoC 기능의 가장 기본이 되는 인터페이스이다.

다만 실제 애플리케이션 개발에서 `BeanFactory`를 직접 사용하는 경우는 많지 않다. 보통은 이를 확장한 `ApplicationContext`를 사용한다.

## 3. ApplicationContext

`ApplicationContext`는 `BeanFactory`를 확장한 IoC 컨테이너이다.

즉, `BeanFactory`의 기능을 모두 포함하면서 실제 애플리케이션 개발에 필요한 부가 기능을 추가로 제공한다.

```java
ApplicationContext context =
        new AnnotationConfigApplicationContext(AppConfig.class);

UserService userService = context.getBean(UserService.class);
```

`ApplicationContext`가 제공하는 대표적인 기능은 다음과 같다.

* 빈 생성과 의존관계 주입
* 빈 조회
* 메시지 소스 처리
* 이벤트 발행
* 리소스 로딩
* 환경 정보 관리
* 국제화 지원
* 빈 후처리기 적용
* AOP 적용 지원

따라서 `BeanFactory`가 스프링 IoC의 핵심 기능을 제공하는 기본 컨테이너라면, `ApplicationContext`는 실제 애플리케이션 운영에 필요한 기능까지 포함한 확장 컨테이너라고 볼 수 있다.

실무에서 일반적으로 “스프링 컨테이너”라고 하면 대부분 `ApplicationContext`를 의미한다.

| 구분    | BeanFactory         | ApplicationContext            |
| ----- | ------------------- | ----------------------------- |
| 역할    | 기본 IoC 컨테이너         | 확장된 IoC 컨테이너                  |
| 주요 기능 | 빈 생성, 의존관계 설정, 빈 조회 | BeanFactory 기능 + 애플리케이션 부가 기능 |
| 사용 위치 | 스프링 내부 구조 이해에 중요    | 실제 애플리케이션에서 주로 사용             |
| 특징    | 핵심 기능 중심            | 메시지, 이벤트, 리소스, 환경 정보 등 제공     |


## 4. 설정 메타정보와 BeanDefinition

스프링 컨테이너는 아무 정보 없이 객체를 생성하지 않는다.

컨테이너가 빈을 생성하려면 어떤 클래스를 빈으로 등록할지, 어떤 이름을 사용할지, 어떤 의존관계를 주입할지 알아야 한다.

이 정보를 설정 메타정보라고 한다.

설정 메타정보에는 다음과 같은 내용이 포함된다.

* 빈으로 등록할 클래스
* 빈 이름
* 빈 스코프
* 의존관계 정보
* 초기화 메서드
* 종료 메서드
* 빈 생성 방식

스프링은 다양한 방식으로 설정 메타정보를 제공받을 수 있다.

XML 기반 설정을 사용할 수도 있다.

```xml
<bean id="userService" class="com.example.UserService">
    <constructor-arg ref="userRepository"/>
</bean>
```

자바 설정을 사용할 수도 있다.

```java
@Configuration
public class AppConfig {

    @Bean
    public UserService userService(UserRepository userRepository) {
        return new UserService(userRepository);
    }
}
```

애노테이션 기반 설정을 사용할 수도 있다.

```java
@Service
public class UserService {
}
```

중요한 점은 설정 방식이 XML이든, 자바 코드이든, 애노테이션이든 상관없이 스프링 내부에서는 이를 `BeanDefinition`이라는 빈 메타정보로 변환해서 사용한다는 것이다.

즉, 스프링 컨테이너는 클래스 자체를 바로 보고 객체를 만드는 것이 아니라, 빈 메타정보를 기준으로 빈을 생성하고 관리한다.

## 5. IoC 컨테이너의 동작 흐름

스프링 IoC 컨테이너의 기본 동작 흐름은 다음과 같다.

```text
설정 메타정보 읽기
        ↓
BeanDefinition 생성 및 등록
        ↓
빈 오브젝트 생성
        ↓
의존관계 주입
        ↓
초기화 작업 수행
        ↓
애플리케이션에서 빈 사용
        ↓
컨테이너 종료 시 소멸 작업 수행
```

개발자는 객체를 직접 생성하고 연결하는 대신, 설정 정보를 제공한다.

컨테이너는 그 설정 정보를 바탕으로 객체를 만들고 의존관계를 연결한다.

이 구조 덕분에 객체 생성과 의존관계 관리가 애플리케이션 코드에서 분리된다.

## 6. ApplicationContext 구현체

`ApplicationContext`는 인터페이스이므로, 설정 방식과 실행 환경에 따라 여러 구현체를 사용할 수 있다.


### 1) AnnotationConfigApplicationContext

`AnnotationConfigApplicationContext`는 자바 설정 클래스와 애노테이션 기반 설정을 읽어서 컨테이너를 생성한다.

```java
ApplicationContext context =
        new AnnotationConfigApplicationContext(AppConfig.class);
```

```java
@Configuration
public class AppConfig {

    @Bean
    public UserService userService() {
        return new UserService();
    }
}
```

또는 컴포넌트 스캔을 함께 사용할 수 있다.

```java
@Configuration
@ComponentScan("com.example")
public class AppConfig {
}
```

현재 스프링 부트나 최신 스프링 애플리케이션에서는 XML보다 자바 설정과 애노테이션 기반 설정을 더 많이 사용한다.

### 2) GenericApplicationContext

`GenericApplicationContext`는 특정 설정 방식에 종속되지 않는 범용 `ApplicationContext` 구현체이다.

`ClassPathXmlApplicationContext`는 XML을 클래스패스에서 읽는 방식이고, `AnnotationConfigApplicationContext`는 자바 설정과 애노테이션을 읽는 방식이다.

반면 `GenericApplicationContext`는 특정 설정 포맷을 전제로 하지 않는다.

내부적으로 `DefaultListableBeanFactory`를 가지고 있고, 여기에 빈 메타정보를 직접 등록하거나 별도의 `BeanDefinitionReader`를 붙여 사용할 수 있다.

```java
GenericApplicationContext context = new GenericApplicationContext();

RootBeanDefinition beanDefinition =
        new RootBeanDefinition(UserService.class);

context.registerBeanDefinition("userService", beanDefinition);

context.refresh();

UserService userService = context.getBean(UserService.class);

context.close();
```

여기서 중요한 점은 `refresh()`를 직접 호출해야 한다는 것이다.

`ClassPathXmlApplicationContext`나 `AnnotationConfigApplicationContext`는 생성 과정에서 설정 정보를 읽고 컨테이너 초기화까지 진행한다.

하지만 `GenericApplicationContext`는 빈 메타정보를 직접 등록할 수 있는 범용 컨테이너이기 때문에, 필요한 빈 정의를 등록한 뒤 `refresh()`를 호출해서 컨테이너를 초기화해야 한다.

일반 애플리케이션 개발보다는 스프링 내부 구조 이해, 테스트, 프레임워크 확장 코드에서 의미가 있다.

실제로 JUnit테스트에서 테스트 내에서 사용할 수 있는 애플리케이션 컨텍스트를 해당 구현체로 만든다고 한다.

### 3) WebApplicationContext

`WebApplicationContext`는 웹 애플리케이션을 위한 `ApplicationContext`이다.

일반 `ApplicationContext`가 독립 실행형 애플리케이션의 빈을 관리하는 컨테이너라면, `WebApplicationContext`는 서블릿 기반 웹 애플리케이션에서 사용되는 컨테이너이다.

웹 애플리케이션에서는 단순히 서비스, 리포지토리 같은 빈만 관리하면 되는 것이 아니다.

웹 환경과 연결되어야 하기 때문에 다음과 같은 정보가 필요하다.

* `ServletContext`
* 서블릿 설정
* 웹 애플리케이션 루트 경로
* 웹 계층 전용 빈
* `DispatcherServlet`과의 연동
* 요청 처리에 필요한 Spring MVC 관련 빈

그래서 Spring MVC 환경에서는 일반 `ApplicationContext`보다 웹 환경에 맞게 확장된 `WebApplicationContext`가 사용된다.

## 7. Root WebApplicationContext와 Servlet WebApplicationContext

전통적인 Spring MVC 구조에서는 컨텍스트가 계층적으로 구성될 수 있다.

```text
Root WebApplicationContext
        ├── Service
        ├── Repository
        ├── DataSource
        └── TransactionManager

DispatcherServlet WebApplicationContext
        ├── Controller
        ├── HandlerMapping
        ├── HandlerAdapter
        ├── ViewResolver
        └── Web MVC Bean
```

### 1) Root WebApplicationContext

`Root WebApplicationContext`는 애플리케이션 전체에서 공유되는 루트 컨테이너이다.

보통 다음과 같은 빈을 둔다.

* Service
* Repository
* DataSource
* TransactionManager
* Security 관련 공통 빈
* 비즈니스 계층과 인프라 계층 빈

### 2) DispatcherServlet WebApplicationContext

`DispatcherServlet`도 자신만의 `WebApplicationContext`를 가질 수 있다.

여기에는 보통 웹 MVC 처리에 필요한 빈이 들어간다.

* Controller
* HandlerMapping
* HandlerAdapter
* ViewResolver
* Converter
* Formatter
* Interceptor

`DispatcherServlet WebApplicationContext`는 `Root WebApplicationContext`를 부모 컨텍스트로 참조할 수 있다.

즉, 웹 계층 컨텍스트에서는 루트 컨텍스트의 서비스 빈을 사용할 수 있다.

반대로 루트 컨텍스트가 웹 계층의 컨트롤러 빈에 의존하는 구조는 일반적이지 않다.

이 구조는 웹 계층과 비즈니스 계층을 분리하기 위한 전통적인 Spring MVC 구성 방식이다.

## 8. WebApplicationContext 구현체

`WebApplicationContext`에도 여러 구현체가 있다.

### 1) XmlWebApplicationContext

`XmlWebApplicationContext`는 XML 설정을 기반으로 웹 애플리케이션 컨텍스트를 구성하는 방식이다.

전통적인 Spring MVC에서 자주 사용되던 방식이다.

```xml
<context-param>
    <param-name>contextConfigLocation</param-name>
    <param-value>/WEB-INF/applicationContext.xml</param-value>
</context-param>
```

### 2) AnnotationConfigWebApplicationContext

`AnnotationConfigWebApplicationContext`는 자바 설정 클래스와 애노테이션 기반 설정을 사용하는 웹 애플리케이션 컨텍스트이다.

XML 대신 `@Configuration`, `@ComponentScan`, `@EnableWebMvc` 등을 사용한다.

```java
AnnotationConfigWebApplicationContext context =
        new AnnotationConfigWebApplicationContext();

context.register(WebConfig.class);
```

### 3) GenericWebApplicationContext

`GenericWebApplicationContext`는 `GenericApplicationContext`의 웹 환경용 하위 클래스이다.

개념적으로는 다음과 같다.

```text
GenericApplicationContext
        ↑
GenericWebApplicationContext
```

`GenericApplicationContext`가 범용 컨테이너라면, `GenericWebApplicationContext`는 여기에 웹 환경, 특히 `ServletContext`와 연결될 수 있는 기능이 추가된 컨테이너라고 볼 수 있다.

## 9. 일반 Java 애플리케이션에서의 실행 흐름

일반 Java 애플리케이션은 `main()` 메서드가 시작점이다.

```java
public static void main(String[] args) {
    ApplicationContext context =
            new AnnotationConfigApplicationContext(AppConfig.class);

    MyApplication app = context.getBean(MyApplication.class);
    app.run();
}
```

흐름은 다음과 같다.

```text
개발자가 main() 실행
        ↓
ApplicationContext 생성
        ↓
빈 생성
        ↓
DI 수행
        ↓
시작용 빈 getBean()
        ↓
run() 호출
        ↓
애플리케이션 로직 실행
```

이 경우 `main()`은 애플리케이션의 진입점이다.

스프링 컨테이너는 빈을 만들고 DI해두지만, 어떤 빈을 꺼내서 어떤 메서드를 호출할지는 `main()`에서 정한다.

그래서 책에서 “IoC 컨테이너는 최초로 애플리케이션을 가동할 빈 하나를 제공해주는 것까지”라고 설명한 것이다.

## 10. 전통적인 웹 애플리케이션에서의 실행 흐름

전통적인 Spring MVC 웹 애플리케이션은 보통 `.war` 파일로 만들어 외부 Tomcat 같은 서블릿 컨테이너에 배포했다.

이 방식에서는 개발자가 작성한 `main()` 메서드가 없다.

```text
외부 Tomcat 실행
        ↓
WAR 배포
        ↓
Tomcat이 웹 애플리케이션 로딩
        ↓
Spring WebApplicationContext 생성
        ↓
DispatcherServlet 초기화
        ↓
HTTP 요청 대기
```

브라우저에서 요청이 오면 서블릿 컨테이너가 요청을 받는다.

```text
브라우저 HTTP 요청
        ↓
Tomcat
        ↓
요청 URL 확인
        ↓
매핑된 서블릿 찾기
        ↓
DispatcherServlet 실행
        ↓
Controller 호출
        ↓
Service 호출
        ↓
Repository 호출
```

책에서 말하는 다음 문장은 이 구조를 설명하는 것이다.

> 웹환경에서는 main() 메소드 대신 서블릿 컨테이너가 브라우저로부터 오는 HTTP 요청을 받아서 해당 요청에 매핑되어 있는 서블릿을 실행해주는 방식으로 동작한다. 서블릿이 일종의 main() 메소드와 같은 역할을 한다.

이 말은 웹 애플리케이션에서는 개발자가 만든 `main()`이 프로그램 흐름을 시작하지 않는다는 뜻이다.

대신 Tomcat 같은 서블릿 컨테이너가 HTTP 요청을 받고, 해당 요청에 맞는 서블릿을 실행한다.

Spring MVC에서는 그 서블릿이 보통 `DispatcherServlet`이다.

즉, 웹 요청 하나의 관점에서 보면 애플리케이션 로직으로 들어가는 입구는 `main()`이 아니라 `DispatcherServlet`이다.

## 11. 서블릿이 main() 같은 역할을 한다는 의미

일반 Java 애플리케이션에서는 `main()`이 한 번 실행되면서 애플리케이션이 시작된다.

```text
main()
  ↓
서비스 호출
  ↓
비즈니스 로직 실행
```

반면 웹 애플리케이션에서는 사용자의 요청이 들어올 때마다 서블릿이 실행된다.

```text
HTTP 요청
  ↓
Servlet
  ↓
Controller
  ↓
Service
  ↓
비즈니스 로직 실행
```

그래서 서블릿이 `main()`과 같은 역할을 한다는 것은, 서블릿이 애플리케이션의 전체 시작점이라는 뜻이라기보다 **요청 단위의 실행 흐름을 시작하는 진입점**이라는 뜻이다.

정확히는 서블릿의 `service()` 메서드가 요청마다 실행된다.

```java
protected void service(HttpServletRequest request, HttpServletResponse response) {
    // 요청 처리 시작
}
```

Spring MVC에서는 이 역할을 `DispatcherServlet`이 담당한다.

```text
HTTP 요청
   ↓
Tomcat
   ↓
DispatcherServlet
   ↓
HandlerMapping
   ↓
Controller
   ↓
Service
   ↓
Repository
```

따라서 웹 애플리케이션에서는 `main()`이 아니라 서블릿이 요청 처리의 시작점 역할을 한다고 볼 수 있다.

## 12. Spring Boot에서는 main()이 있다

현재 사용하는 Spring Boot 프로젝트에는 분명히 `main()`이 있다.

```java
@SpringBootApplication
public class EmotionDiarySpringApplication {

    public static void main(String[] args) {
        SpringApplication.run(EmotionDiarySpringApplication.class, args);
    }
}
```

그리고 개발자는 IntelliJ 실행 버튼을 누르거나, 터미널에서 다음 명령어로 직접 실행한다.

```bash
java -jar emotion-diary.jar
```

그러면 `main()`이 실행된다.

따라서 “Spring Boot에서는 개발자가 main을 직접 구동시키는 것 아닌가?”라는 이해는 맞다.

다만 Spring Boot의 `main()`은 비즈니스 로직을 직접 실행하는 일반적인 `main()`과는 역할이 다르다.

Spring Boot의 `main()`은 다음 일을 한다.

```text
main() 실행
        ↓
SpringApplication.run()
        ↓
ApplicationContext 생성
        ↓
빈 생성 및 DI
        ↓
내장 Tomcat 실행
        ↓
DispatcherServlet 등록
        ↓
HTTP 요청 대기
```

즉, Spring Boot에서 `main()`은 애플리케이션을 시작하고 내장 서블릿 컨테이너를 띄우는 역할을 한다.

그 이후 실제 웹 요청 처리는 여전히 서블릿 컨테이너와 `DispatcherServlet`을 통해 진행된다.

## 13. Spring Boot에서 사용하는 ApplicationContext

다음 코드는 개발자가 직접 `ApplicationContext` 구현체를 선택하지 않는다.

```java
public static void main(String[] args) {
    SpringApplication.run(EmotionDiarySpringApplication.class, args);
}
```

`SpringApplication.run()`이 현재 애플리케이션의 종류를 판단해서 적절한 `ApplicationContext`를 생성한다.

대표적으로 다음과 같이 나뉜다.

| 애플리케이션 종류               | 주로 사용되는 ApplicationContext                            |
| ----------------------- | ----------------------------------------------------- |
| 일반 Java 애플리케이션          | `AnnotationConfigApplicationContext` 계열               |
| Servlet 기반 웹 애플리케이션     | `AnnotationConfigServletWebServerApplicationContext`  |
| Reactive WebFlux 애플리케이션 | `AnnotationConfigReactiveWebServerApplicationContext` |

현재 프로젝트처럼 `spring-boot-starter-web`을 사용하는 Spring MVC 기반 애플리케이션이라면 보통 `AnnotationConfigServletWebServerApplicationContext`가 사용된다.

이 이름을 나눠보면 의미가 분명하다.

```text
AnnotationConfig
+ Servlet
+ WebServer
+ ApplicationContext
```

각 의미는 다음과 같다.

* `AnnotationConfig`: 자바 설정과 애노테이션 기반 설정을 읽는다.
* `Servlet`: 서블릿 기반 웹 애플리케이션이다.
* `WebServer`: 내장 Tomcat 같은 웹 서버를 실행한다.
* `ApplicationContext`: 스프링 IoC 컨테이너이다.

즉, Spring Boot의 일반적인 MVC 웹 프로젝트에서는 `AnnotationConfigServletWebServerApplicationContext`가 생성된다고 보면 된다.

## 14. Spring Boot 실행 흐름

```text
IntelliJ 실행 버튼 또는 java -jar 실행
        ↓
main() 실행
        ↓
SpringApplication.run()
        ↓
애플리케이션 타입 판단
        ↓
Servlet 웹 애플리케이션으로 판단
        ↓
AnnotationConfigServletWebServerApplicationContext 생성
        ↓
자동 설정 적용
        ↓
컴포넌트 스캔
        ↓
BeanDefinition 등록
        ↓
빈 생성 및 DI
        ↓
내장 Tomcat 실행
        ↓
DispatcherServlet 등록
        ↓
HTTP 요청 대기
```

그 후 브라우저에서 요청이 들어오면 다음 흐름으로 처리된다.

```text
브라우저 HTTP 요청
        ↓
내장 Tomcat
        ↓
DispatcherServlet
        ↓
HandlerMapping
        ↓
Controller
        ↓
Service
        ↓
Repository
```

따라서 Spring Boot에서는 다음처럼 구분하면 된다.

```text
애플리케이션 실행 시작점: main()
웹 요청 처리 시작점: DispatcherServlet
```

Spring Boot에서는 개발자가 `main()`을 직접 실행하는 것이 맞다.

하지만 웹 요청 하나하나의 처리 관점에서는 여전히 `DispatcherServlet`이 진입점이다.
