# 2.1 공통 개념

스프링이 지원하는 데이터 액세스 기술을 하나씩 살펴보기 전에, 모든 기술에 공통적으로 적용되는 원칙과 기본 개념을 정리한다.


## 2.1.1 DAO 패턴

데이터 액세스 계층은 **DAO 패턴**으로 분리하는 것이 원칙. 비즈니스 로직이 단순하면 통합해도 되지만, 의미 있는 비즈니스 로직이 있는 엔터프라이즈 애플리케이션이라면 분리해야 한다.

### 정의
- **DTO 또는 도메인 오브젝트만** 사용하는 인터페이스로 외부와 통신.
- DAO는 **구현 기술 정보를 외부에 노출하지 않는다.**

### 효과
- 데이터 액세스 기술 변경/혼용해도 사용 코드에 영향 없음.
- 서비스 계층을 **순수 POJO**로 유지 가능.
- DAO 인터페이스가 단순 DTO/도메인만 다루므로 **목 오브젝트로 단위 테스트 작성 용이**.


### DAO 인터페이스와 DI

원칙: 인터페이스로 접근, DI로 주입. 인터페이스에 구체적인 데이터 액세스 기술 관련 API/정보를 노출하지 않는다.

지킬 점:
- DAO 클래스의 모든 public 메소드를 **습관적으로 인터페이스에 추가하지 말 것.** 서비스 계층 입장에서 의미 있는 메소드만 공개.
- DI용 수정자 메소드(`setDataSource()` 등)는 public이라도 **기계적으로 인터페이스에 추가하지 말 것.** (DI 설정용이지 클라이언트가 호출할 메소드가 아님)
- **특정 기술에서만 의미 있는 메소드명**은 피한다.
  - 예) JPA의 `persist()`, `merge()`를 그대로 DAO 메소드명으로 쓰지 말 것. 일반적인 `add()`, `update()` 사용.


### 예외처리

기본 원칙:
- 데이터 액세스 예외는 **대부분 복구 불가**. DAO 밖으로 던질 때는 **런타임 예외**여야 함.
- `throws SQLException` (기술 노출), `throws Exception` (무책임) 둘 다 금지.
- 서비스 계층은 대부분의 DAO 예외를 직접 다룰 필요 없음.

#### 의미 있는 예외 (잡아야 할 때)
- 중복 키 예외, 낙관적 락킹(optimistic locking) 예외 등.
- **문제**: 기술/DB마다 던지는 예외가 제각각이라 서비스 계층이 구현 기술을 알아야 함.

#### 스프링의 해결책
1. **`DataAccessException` 계층** — 기술/DB와 무관한 일관된 예외 추상화.
2. **예외 변환 서비스** — 기술별 예외를 스프링 예외로 변환.
   - 템플릿/콜백 (`JdbcTemplate` 등) 사용 시 자동 적용.
   - API 직접 사용 시 `@Repository` + AOP 예외 전환.

> 최신 데이터 액세스 기술은 자체적으로 런타임 예외를 사용. 일관된 예외 복구가 필요 없으면 굳이 변환 서비스 안 써도 됨.


## 2.1.2 템플릿과 API

데이터 액세스 코드는 대부분 `try/catch/finally`로 리소스 누수를 막느라 **반복적이고 지저분**해진다. 외부 리소스와 연동되어 예외 상황이 다양하기 때문.

### 템플릿/콜백이 해주는 것
스프링이 제공하는 템플릿은 미리 만들어진 작업 흐름을 담아 다음을 해결한다.
- **반복 코드 제거**
- **예외 변환** (기술별 예외를 `DataAccessException`으로)
- **트랜잭션 동기화**

### 템플릿의 단점
- 데이터 액세스 기술의 원래 API 대신 **템플릿 API**를 사용해야 함.
- 콜백을 **익명 내부 클래스**로 작성할 때 가독성이 떨어질 수 있음 (내장 콜백 메소드를 쓰면 대부분 해결).

### 템플릿 vs API 직접 사용
- 일부 기술은 템플릿 대신 **API 직접 사용**도 지원. 해당 기술의 확장 기법 + AOP로 동일 혜택(예외 변환·트랜잭션 동기화) 제공.
- 둘 다 지원하는 기술이면 취향대로 선택.
- 단, **스프링이 지원하지 않는 방식**으로 쓰면 예외 변환/트랜잭션 동기화 혜택을 못 받으므로 피할 것.


## 2.1.3 DataSource

- `Connection`은 모든 데이터 액세스 기술의 **필수 리소스**.
- 엔터프라이즈 환경에서 매번 새 커넥션을 만들면 비효율적이므로, 미리 만들어둔 **커넥션 풀(pool)** 에서 꺼내 쓰고 반환하는 풀링 기법 사용.

### 스프링의 권장
- `DataSource`는 **항상 독립된 빈**으로 등록.
- 데이터 액세스 기술의 자체 설정파일에 DB 연결 정보를 박아두면 안 됨 (정보가 특정 기술 내부에 종속됨).
- 빈으로 등록해 여러 서비스가 공유.

> 다중 사용자를 갖는 엔터프라이즈 시스템에서는 반드시 **풀링 기능을 지원하는 DataSource** 사용.


### 학습/테스트용 DataSource

운영 환경 절대 금지. 개발 중 테스트용 DataSource를 그대로 운영에 적용해 애먹는 경우가 많음. **개발/테스트/운영 설정 파일 분리** 필수.

| 종류 | 특징 |
|---|---|
| `SimpleDriverDataSource` | 호출마다 새 커넥션 생성, 풀 없음. 테스트 전용. |
| `SingleConnectionDataSource` | 단일 커넥션 재사용. 순차 통합 테스트만 OK, 멀티스레드는 위험. 매번 생성 안 하므로 `Simple~`보다 빠름. |


### 운영용 DB 커넥션 풀

보통 **애플리케이션 레벨의 전용 풀**로 사용. 모두 스프링 빈으로 바로 등록 가능한 수정자 메소드를 가진 클래스가 제공됨.

- **아파치 Commons DBCP** — 가장 유명한 오픈소스.
- **c3p0** — JDBC 3.0 스펙을 준수하는 `Connection`/`Statement` 풀.
- **상용 DB 자체 풀** — 빈 등록·프로퍼티 설정만 가능하면 사용 가능.


### JNDI/WAS DB 풀

WAS가 제공하는 풀을 써야 할 때는 **JNDI로 서버의 `DataSource`에 접근**. `<jee:jndi-lookup>` 태그로 가져온 오브젝트를 스프링 빈으로 등록.

```xml
<jee:jndi-lookup id="dataSource" jndi-name="jdbc/DefaultDS" />
```

**문제**: JNDI 빈은 서버 밖에서는 동작하지 않아 테스트 환경에서 활용 어려움.

**해결**:
1. 테스트용 `DataSource`가 정의된 별도 설정 파일 사용.
2. 스프링 테스트 프레임워크의 `SimpleNamingContextBuilder`로 **JVM 레벨 JNDI 바인딩**. 서버가 아닌 환경에서도 JNDI 등록 빈이 동작하게 만들 수 있음.

```java
SimpleNamingContextBuilder builder = new SimpleNamingContextBuilder();
SimpleDriverDataSource ds = new SimpleDriverDataSource(...);
builder.bind("jdbc/DefaultDS", ds);
builder.activate();
```
