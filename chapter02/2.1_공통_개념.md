# 2.1 공통 개념

스프링이 지원하는 데이터 액세스 기술을 하나씩 살펴보기 전에, 모든 기술에 공통적으로 적용되는 원칙과 기본 개념을 정리한다.

## 2.1.1 DAO 패턴

데이터 액세스 계층은 **DAO 패턴**이라 불리는 방식으로 분리하는 것이 원칙이다.

### DAO 패턴의 정의와 효과

DAO 패턴은 **DTO 또는 도메인 오브젝트만을 사용하는 인터페이스**를 통해 데이터 액세스 기술을 외부에 노출하지 않도록 만드는 것이다. 따라서 DAO는 **구현 기술에 대한 정보를 외부에 공개해서는 안 된다.**

이점:

- DAO를 사용하는 코드에 영향을 주지 않고 데이터 액세스 기술을 변경하거나, 하나 이상의 데이터 액세스 기술을 혼합해서 사용할 수 있다.
- **DAO를 이용하는 서비스 계층의 코드를 기술이나 환경에 종속되지 않는 순수한 POJO로 개발**할 수 있다는 것.
- 언제든지 **목 오브젝트 같은 테스트 대역 오브젝트로 대체해서 단위 테스트를 작성**할 수 있다.


### DAO 인터페이스와 DI

DAO는 **인터페이스를 이용해 접근하고 DI**되도록 만들어야 한다. DAO 인터페이스에는 구체적인 데이터 액세스 기술과 관련된 어떤 API나 정보도 노출하지 않는다.

지킬 점:

- 인터페이스를 만들 때 **DAO 클래스의 모든 public 메소드를 습관적으로 추가하지 않도록** 주의한다.
  - DAO를 사용하는 서비스 계층 코드 입장에서 의미 있는 메소드만 인터페이스로 공개해야 한다.
- DAO 클래스에 DI를 위해 넣은 `setDataSource()` 같은 **수정자 메소드는 public이라 해도 기계적으로 인터페이스에 추가하지 않는다.** (DI 설정용이지 클라이언트가 호출할 메소드가
  아니므로)
- **특정 데이터 액세스 기술에서만 의미 있는 메소드 이름은 피한다.** 
  - 예) JPA는 `persist()`, `merge()` 등을 사용한다. 그러나 JPA로 DAO를 만든다고 해서 DAO의 메소드 이름까지 `persist()`, `merge()`로 정하는 것은 바람직하지 않다. 좀 더 일반적인 `add()`, `update()`와 같은 이름 선택.

### 예외처리

데이터 액세스 중에 발생하는 예외는 **대부분 복구할 수 없다.** 따라서 DAO 밖으로 던져질 때는 **런타임 예외**여야 한다.

- DAO 메소드 선언부에 `throws SQLException`과 같이 **내부 기술을 드러내는 예외를 직접 노출해선 안 된다.** + `throws Exception`과 같은 **무책임한 선언**도
- DAO 내부에서 발생하는 예외는 모두 런타임 예외로 전환해야 한다.

#### 의미 있는 예외를 처리해야 하는 경우
- **중복 키 예외**
- **낙관적 락킹(optimistic locking) 예외**

이런 의미 있는 예외를 처리하려고 할 때, JDBC나 각 데이터 액세스 기술에서 던져주는 예외에는 **일관성이 없다**. 그래서 구현 기술에 따라 달라지는 예외를 서비스 계층에서 알아야 한다는 문제가 생긴다.

스프링은 이 문제를 해결하기 위해 두 가지를 제공한다.

1. **데이터 예외 추상화**: 특정 기술이나 DB의 종류에 상관없이 일관된 의미를 갖는 예외 계층(`DataAccessException`).
2. **예외 변환 서비스**: 각 기술과 DB에서 발생하는 예외를 스프링의 데이터 예외로 변환해주는 서비스.
    - `JdbcTemplate` 같은 **템플릿/콜백 기능을 사용하면 변환 서비스가 자동으로 적용**된다.
    - 데이터 액세스 기술의 API를 직접 사용할 때는 스프링에 내장된 **AOP를 이용한 예외 전환 기능**(`@Repository` 등)을 사용한다.

> 최신 데이터 액세스 기술은 JDBC와 달리 **자체적으로 런타임 예외**를 사용한다. 기술에 독립적인 추상 예외로 전환하고 일관된 예외 복구 기능을 적용할 필요가 없다면, 굳이 스프링의 예외 변환 서비스를
> 사용하지 않아도 된다.

## 2.1.2 템플릿과 API

데이터 액세스 기술을 사용하는 코드는 대부분 **`try/catch/finally`와 판에 박힌 반복**되는 코드로 작성되기 쉽다. 데이터 액세스 기술은 애플리케이션 외부의 리소스와 연동하기 때문에 다양한 예외
상황이 발생할 수 있고, 이때 서버의 제한된 리소스에 **누수가 발생하지 않도록 사용한 리소스를 적절히 반환해주는 코드가 반드시 필요**하다. 그 결과 코드가 길어지고 지저분해지기 쉽다.

### 템플릿이 해결해주는 것

스프링은 DI의 응용 패턴인 **템플릿/콜백 패턴**을 이용해 이런 판에 박힌 코드를 피하고 꼭 필요한 바뀌는 내용만 담을 수 있도록 데이터 액세스 기술을 위한 **템플릿**을 제공한다.

미리 만들어진 작업 흐름을 담은 템플릿이 제공하는 것:

- **반복되는 코드 제거**
- **예외 변환** (기술별 체크 예외 → 스프링 `DataAccessException`)
- **트랜잭션 동기화 기능**

### 템플릿의 단점

- 데이터 액세스 기술의 API를 직접 사용하는 대신 **템플릿이 제공하는 API**를 이용해야 한다.
- **콜백 오브젝트를 익명 내부 클래스**로 작성해야 하는 경우에는 코드를 이해하기가 조금 불편할 수 있다.

### 템플릿 대신 API 직접 사용

그래서 스프링은 **일부 데이터 액세스 기술은 템플릿 대신 해당 기술의 API를 그대로 사용하게 해주기도** 한다. 해당 기술이 제공하는 확장 기법과 AOP 등을 이용해 예외 변환과 트랜잭션 동기화를 제공해줄 수
있기 때문이다.

## 2.1.3 DataSource

- `Connection` = 모든 데이터 액세스의 필수 리소스.
- 매 요청마다 새로 만드는 건 비효율 → **커넥션 풀(pool)** 사용.

### 스프링 권장

- `DataSource`는 **항상 독립된 빈**으로 등록.
- 데이터 액세스 기술 자체 설정파일에 DB 정보를 박아두면 안된다. (특정 기술에 종속됨).
- 스프링 빈으로 등록 → 여러 서비스가 공유.

> 다중 사용자 엔터프라이즈 환경 = 반드시 **풀링 DataSource** 사용.

### 학습/테스트용 DataSource

운영 환경에서 절대 금지. **개발/테스트/운영 설정 파일 분리** 필수.

| 종류                           | 특징                                                       |
|------------------------------|----------------------------------------------------------|
| `SimpleDriverDataSource`     | 호출 때마다 새 커넥션 생성, 풀 없음. 테스트 전용.                           |
| `SingleConnectionDataSource` | 단일 커넥션 재사용. 순차 통합 테스트만 OK, 멀티스레드 지원 안된다. `Simple~`보다 빠름. |

### 오픈소스 / 상용 DB 커넥션 풀 (운영용)

- 보통 애플리케이션 레벨의 전용 풀로 사용. 모두 스프링 빈 등록 가능.
- **DBCP** - 가장 유명한 오픈소스.
- **c3p0** - JDBC 3.0 스펙 준수.
- **상용 DB 자체 풀** — 빈 등록·프로퍼티 설정만 되면 OK.

### JNDI/WAS DB 풀

- WAS가 제공하는 풀을 써야 하는 경우.
- `<jee:jndi-lookup>`으로 가져와 빈 등록.

```xml

<jee:jndi-lookup id="dataSource" jndi-name="jdbc/DefaultDS"/>
```

- 문제: 서버 밖에서는 안 돌아감 → 테스트 불편.
- 해결:
    1. 테스트용 설정 파일 분리.
    2. `SimpleNamingContextBuilder`로 JVM 레벨 JNDI 바인딩.
    ```java
    SimpleNamingContextBuilder builder = new SimpleNamingContextBuilder();
    SimpleDriverDataSource ds = new SimpleDriverDataSource(...);
            builder.
    
    bind("jdbc/DefaultDS",ds);
    builder.
    
    activate();
    ```
