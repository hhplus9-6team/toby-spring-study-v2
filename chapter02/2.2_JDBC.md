# 2.2 JDBC

## 2.2.1 스프링 JDBC 기술과 동작원리

### 스프링의 JDBC 접근 방법

JDBC(Java Database Connectivity)는 자바에서 데이터베이스에 접근하기 위한 가장 기본적인 기술입니다.
하지만 순수 JDBC를 직접 사용할 경우 반복적인 코드가 매우 많이 발생합니다.

대표적으로 다음과 같은 작업들이 반복됩니다.

- Connection 생성
- PreparedStatement 생성
- SQL 실행
- ResultSet 처리
- 예외 처리
- 자원 반환(close)

예를 들어 순수 JDBC 코드에서는 아래와 같은 형태가 자주 등장합니다.

```java
public User findById(Long id) {

    Connection conn = null;
    PreparedStatement ps = null;
    ResultSet rs = null;

    try {
        conn = dataSource.getConnection();

        ps = conn.prepareStatement(
            "SELECT id, name, age FROM users WHERE id = ?"
        );

        ps.setLong(1, id);

        rs = ps.executeQuery();

        if (rs.next()) {
            return new User(
                rs.getLong("id"),
                rs.getString("name"),
                rs.getInt("age")
            );
        }

        return null;

    } catch (SQLException e) {
        throw new RuntimeException(e);

    } finally {

        try {
            if (rs != null) rs.close();
            if (ps != null) ps.close();
            if (conn != null) conn.close();
        } catch (SQLException e) {
        }
    }
}
```

비즈니스 로직보다 JDBC 관련 부가 코드가 훨씬 많아지는 문제가 발생합니다.

스프링 JDBC는 이러한 반복 작업을 템플릿 메소드 패턴 기반으로 추상화하여 개발자가 SQL과 데이터 매핑에만 집중할 수 있도록 도와줍니다.

대표적인 클래스는 다음과 같습니다.

- JdbcTemplate
- SimpleJdbcTemplate
- NamedParameterJdbcTemplate
- SimpleJdbcInsert
- SimpleJdbcCall

특히 JdbcTemplate 은 스프링 JDBC 기술의 핵심 역할을 수행합니다.

### 스프링 JDBC가 해주는 작업

스프링 JDBC는 개발자를 대신해서 JDBC의 반복적인 작업을 처리해줍니다.

대표적으로 다음과 같은 작업을 자동화합니다.

#### 1. Connection 획득 및 반환

```java
Connection conn = dataSource.getConnection();
```

개발자가 직접 관리하지 않아도 됩니다.

트랜잭션이 적용된 경우에는 현재 트랜잭션과 연결된 Connection을 가져옵니다.

#### 2. PreparedStatement 생성 및 실행

```java
PreparedStatement ps = conn.prepareStatement(sql);
```

이 작업도 내부적으로 처리됩니다.

#### 3. ResultSet 순회 및 예외 처리

SQLException을 스프링의 DataAccessException 계층으로 변환해줍니다.

`catch (SQLException e)` 대신 `throw new DataAccessException(...)` 형태로 일관성 있게 처리할 수 있습니다.

#### 4. 자원 정리(close)

```java
rs.close();
ps.close();
conn.close();
```

자원 정리를 자동으로 수행합니다. 따라서 개발자는 핵심 로직에 집중할 수 있습니다.

예를 들어 JdbcTemplate을 사용하면 동일 기능을 아래처럼 간단히 작성할 수 있습니다.

```java
public User findById(Long id) {

    return jdbcTemplate.queryForObject(
        "SELECT id, name, age FROM users WHERE id = ?",
        (rs, rowNum) -> new User(
            rs.getLong("id"),
            rs.getString("name"),
            rs.getInt("age")
        ),
        id
    );
}
```

JDBC 코드의 대부분이 제거된 것을 확인할 수 있습니다.

## 2.2.2 SimpleJdbcTemplate

`SimpleJdbcTemplate` 은 기존 JdbcTemplate을 좀 더 편리하게 사용할 수 있도록 만든 템플릿 클래스입니다.

제네릭과 가변 인자를 적극 활용하여 코드의 가독성을 높였습니다.

(현재 최신 스프링에서는 deprecated 되었지만, 토비의 스프링 Vol.2에서는 학습 목적으로 매우 중요합니다.)

### SimpleJdbcTemplate 생성

보통 DataSource를 주입받아 생성합니다.

```java
@Repository
public class UserDao {

    private SimpleJdbcTemplate jdbcTemplate;

    @Autowired
    public void setDataSource(DataSource dataSource) {
        this.jdbcTemplate = new SimpleJdbcTemplate(dataSource);
    }
}
```

또는 JdbcTemplate 기반으로 생성할 수도 있습니다.

```java
JdbcTemplate jdbcTemplate = new JdbcTemplate(dataSource);

SimpleJdbcTemplate simpleJdbcTemplate = new SimpleJdbcTemplate(jdbcTemplate);
```

### SQL 파라미터

`SimpleJdbcTemplate`은 가변 인자를 사용하여 SQL 파라미터를 훨씬 간단히 전달할 수 있습니다.

#### 기존 JdbcTemplate

```java
jdbcTemplate.queryForObject(
    sql,
    new Object[]{id},
    Integer.class
);
```

#### SimpleJdbcTemplate

```java
simpleJdbcTemplate.queryForObject(
    sql,
    Integer.class,
    id
);
```

배열 생성이 제거되어 코드가 훨씬 간결해집니다.

### SQL 실행 메소드

INSERT, UPDATE, DELETE 같은 변경 작업에는 `update()` 메소드를 사용합니다.

```java
String sql = "UPDATE users SET age = ? WHERE id = ?";

simpleJdbcTemplate.update(sql, 30, 1L);
```

반환값은 영향받은 row 수입니다.

```java
int affectedRows = simpleJdbcTemplate.update(sql, 30, 1L);
```

### SQL 조회 메소드

#### 단건 조회

```java
String sql = "SELECT count(*) FROM users";

Integer count = simpleJdbcTemplate.queryForObject(
    sql,
    Integer.class
);
```

#### 객체 조회

```java
String sql = "SELECT * FROM users WHERE id = ?";

User user = simpleJdbcTemplate.queryForObject(
    sql,
    (rs, rowNum) -> new User(
        rs.getLong("id"),
        rs.getString("name"),
        rs.getInt("age")
    ),
    1L
);
```


여기서 사용되는 (rs, rowNum) 인터페이스를 `RowMapper` 라고 합니다.

ResultSet의 한 행(row)을 객체로 변환하는 역할을 수행합니다.

#### 목록 조회

```java
String sql = "SELECT * FROM users";

List<User> users = simpleJdbcTemplate.query(
    sql,
    (rs, rowNum) -> new User(
        rs.getLong("id"),
        rs.getString("name"),
        rs.getInt("age")
    )
);
```

### SQL 배치 메소드

배치 작업은 여러 SQL을 한 번에 실행할 때 사용합니다.

```java
String sql = "INSERT INTO users(name, age) VALUES(?, ?)";

List<Object[]> batchArgs = List.of(
    new Object[]{"jun", 29},
    new Object[]{"kim", 31},
    new Object[]{"lee", 25}
);

jdbcTemplate.batchUpdate(sql, batchArgs);
```


배치 처리를 사용하면 DB round trip 횟수를 줄여 성능을 개선할 수 있습니다.

대량 데이터 처리 시 매우 중요합니다.

## 2.2.3 SimpleJdbcInsert

`SimpleJdbcInsert` 는 INSERT SQL을 직접 작성하지 않고도 데이터를 저장할 수 있도록 도와주는 클래스입니다.

테이블 메타데이터를 기반으로 자동으로 INSERT SQL을 생성합니다.

### SimpleJdbcInsert 생성

```java
SimpleJdbcInsert jdbcInsert = new SimpleJdbcInsert(dataSource).withTableName("users");
```

기본적으로 테이블명을 지정합니다.

### 자동 생성 키 설정

```java
SimpleJdbcInsert jdbcInsert = 
        new SimpleJdbcInsert(dataSource).withTableName("users")
                .usingGeneratedKeyColumns("id");
```

자동 증가 PK를 반환받을 수 있습니다.

### SimpleJdbcInsert 실행

파라미터는 Map 형태로 전달합니다.

```java
Map<String, Object> params = new HashMap<>();

params.put("name", "jun");
params.put("age", 29);

Number key =
jdbcInsert.executeAndReturnKey(params);

System.out.println(key.longValue());
```

실제로는 아래와 같은 SQL이 자동 생성됩니다.

```sql
INSERT INTO users(name, age)
VALUES (?, ?)
```

SQL을 직접 작성하지 않아도 되는 장점이 있습니다.

## 2.2.4 SimpleJdbcCall

`SimpleJdbcCall` 은 저장 프로시저(Stored Procedure) 또는 함수(Function)를 호출하기 위한 클래스입니다.

JDBC의 CallableStatement를 추상화합니다.

### SimpleJdbcCall 생성

```java
SimpleJdbcCall jdbcCall = new SimpleJdbcCall(dataSource).withProcedureName("find_user_count");
```

저장 프로시저 이름을 지정합니다.

### SimpleJdbcCall 실행

#### 입력 파라미터 전달

```java
SqlParameterSource params = 
        new MapSqlParameterSource()
                .addValue("age", 20);

Map<String, Object> result = jdbcCall.execute(params);
```

#### 결과 조회

```java
Integer count = (Integer) result.get("user_count");
```

예를 들어 DB 프로시저가 아래와 같다고 가정해봅시다.

```sql
CREATE PROCEDURE find_user_count(
    IN age INT,
    OUT user_count INT
)
```

스프링은 내부적으로 CallableStatement를 생성하여 프로시저 호출을 수행합니다.

## 2.2.5 스프링 JDBC DAO

스프링 JDBC DAO의 핵심 목적은 다음과 같습니다.

- JDBC 반복 코드 제거
- DAO 책임 분리
- SQL 실행과 객체 매핑 단순화
- 예외 처리 일관성 확보

보통 DAO는 다음과 같은 형태로 작성됩니다.

```java
@Repository
public class UserDao {

    private final JdbcTemplate jdbcTemplate;

    public UserDao(DataSource dataSource) {
        this.jdbcTemplate =
            new JdbcTemplate(dataSource);
    }

    public void save(User user) {

        String sql =
            "INSERT INTO users(name, age) VALUES(?, ?)";

        jdbcTemplate.update(
            sql,
            user.getName(),
            user.getAge()
        );
    }

    public User findById(Long id) {

        String sql =
            "SELECT * FROM users WHERE id = ?";

        return jdbcTemplate.queryForObject(
            sql,
            this::mapUser,
            id
        );
    }

    private User mapUser(ResultSet rs, int rowNum)
        throws SQLException {

        return new User(
            rs.getLong("id"),
            rs.getString("name"),
            rs.getInt("age")
        );
    }
}
```

여기서 중요한 점은 다음과 같습니다.

- SQL 실행 로직은 JdbcTemplate이 담당
- DAO는 비즈니스 관점의 데이터 접근 책임만 담당
- RowMapper를 재사용하여 객체 매핑 중복 제거
- SQLException 대신 DataAccessException 사용

즉, 스프링 JDBC는 “반복되는 JDBC 인프라 코드”를 프레임워크가 대신 처리하고, 개발자는 데이터 접근 로직 자체에 집중할 수 있도록 설계된 기술이라고 볼 수 있습니다.

## (+) 왜 SimpleJdbcTemplate 은 deprecated 되었을까요?

SimpleJdbcTemplate 은 스프링 3.1부터 deprecated 되었고, 이후 제거되었습니다.
이유는 크게 보면 JdbcTemplate 자체가 충분히 좋아졌기 때문입니다.

초기 JdbcTemplate 은 다음과 같은 불편함이 있었습니다.

```java
jdbcTemplate.queryForObject(
    sql,
    new Object[]{id},
    Integer.class
);
```

- 파라미터 배열을 직접 만들어야 함
- 제네릭 지원이 부족함
- 타입 안정성이 떨어짐

그래서 이를 개선하기 위해 등장한 것이 SimpleJdbcTemplate 이었습니다.

```java
simpleJdbcTemplate.queryForObject(
    sql,
    Integer.class,
    id
);
```

훨씬 간결하고 가독성이 좋아졌습니다.

하지만 이후 자바와 스프링 버전이 올라가면서 상황이 달라졌습니다.

특히 다음 요소들이 영향을 주었습니다.

- Java 5 제네릭 지원 강화
- 가변 인자(varargs) 지원
- JdbcTemplate 기능 개선
- 람다 표현식 등장(Java 8)

결국 기존 JdbcTemplate 도 아래처럼 사용할 수 있게 되었습니다.

```java
jdbcTemplate.queryForObject(
    sql,
    Integer.class,
    id
);
```


즉, SimpleJdbcTemplate 의 장점이 더 이상 특별하지 않게 된 것입니다.

그래서 스프링 팀은 기능이 겹치는 두 클래스를 유지하기보다 JdbcTemplate 하나로 통합하는 방향을 선택했습니다.

현재는 보통 아래 조합을 많이 사용합니다.

- JdbcTemplate
- NamedParameterJdbcTemplate
- RowMapper
- BeanPropertyRowMapper
