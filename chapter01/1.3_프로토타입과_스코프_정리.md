# 1.3 프로토타입과 스코프 정리

## 서론

스프링 빈은 기본적으로 싱글톤으로 관리됩니다.
즉, 컨테이너가 생성한 하나의 객체를 애플리케이션 전체에서 공유해서 사용합니다.

하지만 모든 객체가 싱글톤으로 동작해야 하는 것은 아닙니다.
어떤 객체는 요청마다 새롭게 생성되어야 하고, 어떤 객체는 특정 요청 범위 안에서만 유지되어야 합니다.

이번 장에서는 이러한 문제를 해결하기 위한 프로토타입 스코프와 다양한 스코프 전략에 대해 설명합니다.

## 1.3.1 프로토타입 스코프

### 프로토타입 빈의 생명주기와 종속성

스프링 빈은 기본적으로 싱글톤 스코프를 사용합니다.

```java
@Component
public class OrderService {
}
```

이 경우 스프링 컨테이너는 OrderService 객체를 하나만 생성하고 계속 재사용합니다.

반면 프로토타입 스코프는 요청할 때마다 새로운 객체를 생성합니다.

```java
@Component
@Scope("prototype")
public class OrderContext {
}
```

프로토타입 빈은 `getBean()`을 호출할 때마다 새로운 객체가 생성됩니다.

```java
OrderContext context1 = applicationContext.getBean(OrderContext.class);
OrderContext context2 = applicationContext.getBean(OrderContext.class);

System.out.println(context1 == context2);
```


결과는 `false` 로 나타납니다. 즉, 매번 다른 객체가 생성됩니다.

싱글톤과 프로토타입의 가장 큰 차이는 생명주기 관리 방식입니다.

싱글톤 빈은 컨테이너가 생성부터 소멸까지 모두 관리합니다.

```
생성 → 초기화 → 사용 → destroy
```

반면 프로토타입 빈은 생성 후 컨테이너가 더 이상 관리하지 않습니다. 즉, 프로토타입 빈은 destroy 메서드가 호출되지 않습니다.

### 프로토타입 빈의 용도

프로토타입 스코프는 상태를 가지는 객체를 생성할 때 사용할 수 있습니다.

예를 들어 주문 처리 과정에서 요청마다 별도의 상태를 저장해야 하는 경우를 생각해봅시다.

```java
@Component
@Scope("prototype")
public class OrderContext {

    private int totalPrice;

    public void addPrice(int price) {
        totalPrice += price;
    }

    public int getTotalPrice() {
        return totalPrice;
    }
}
```

이 객체는 주문 처리 중에만 사용되는 임시 상태를 가집니다. 이렇게 하면 요청마다 독립적인 상태를 유지할 수 있습니다.

### DI와 DL

프로토타입 빈을 사용할 때 가장 중요한 문제는 “언제 객체를 생성할 것인가”입니다.

보통 스프링은 DI(Dependency Injection)를 사용합니다.

```java
@RequiredArgsConstructor
@Service
public class OrderService {

    private final OrderContext orderContext;
}
```

하지만 이 방식에는 문제가 있습니다.

OrderService는 싱글톤 빈입니다. 싱글톤 빈은 생성 시점에 딱 한 번만 의존 객체를 주입받습니다.

즉, 프로토타입 빈이라 하더라도 실제로는 하나의 객체만 계속 사용하게 됩니다.

프로토타입 빈을 주입받았지만 실제로는 싱글톤처럼 동작하게 됩니다.

이 문제를 해결하기 위해 사용하는 방식이 DL(Dependency Lookup)입니다.

DL은 필요한 시점에 직접 빈을 조회하는 방식입니다.

```java
@Component
@RequiredArgsConstructor
public class OrderService {

    private final ApplicationContext applicationContext;

    public void order() {
        OrderContext context = applicationContext.getBean(OrderContext.class);
    }
}
```

이제 order()가 호출될 때마다 새로운 객체를 가져올 수 있습니다.

### 프로토타입 빈의 DL 전략

하지만 ApplicationContext를 직접 사용하는 방식은 코드가 스프링 컨테이너에 강하게 의존하게 됩니다.

그래서 스프링은 이를 해결하기 위한 여러 전략을 제공합니다.

#### ApplicationContext#getBean()

가장 기본적인 방식은 ApplicationContext에서 직접 빈을 조회하는 것입니다.

```java
@Component
@RequiredArgsConstructor
public class OrderService {

    private final ApplicationContext applicationContext;

    public void order() {
        OrderContext context =
                applicationContext.getBean(OrderContext.class);
    }
}
```

이 방식은 가장 단순하지만, 비즈니스 코드가 스프링 컨테이너에 직접 의존하게 됩니다.

#### BeanFactory#getBean()

ApplicationContext의 상위 인터페이스인 BeanFactory를 사용할 수도 있습니다.

```java
@Component
@RequiredArgsConstructor
public class OrderService {

    private final BeanFactory beanFactory;

    public void order() {
        OrderContext context = beanFactory.getBean(OrderContext.class);
    }
}
```

동작은 거의 동일하지만, 좀 더 최소 기능만 제공하는 컨테이너 인터페이스를 사용한다는 차이가 있습니다.

다만 실무에서는 ApplicationContext를 더 많이 사용합니다.

#### ObjectFactory#getObject()

스프링은 보다 느슨한 DL을 위해 ObjectFactory를 제공합니다.

컨테이너 자체가 아니라 필요한 순간 객체를 꺼내오는 용도의 인터페이스입니다.

```java
@Component
@RequiredArgsConstructor
public class OrderService {

    private final ObjectFactory<OrderContext> objectFactory;

    public void order() {
        OrderContext context = objectFactory.getObject();
    }
}
```

- 단순한 DL 기능 제공
- lazy 조회 가능
- getObject() 호출 시 실제 객체 조회

#### ObjectProvider#getObject()

ObjectProvider는 ObjectFactory를 확장한 인터페이스입니다.

```java
@Component
@RequiredArgsConstructor
public class OrderService {

    private final ObjectProvider<OrderContext> provider;

    public void order() {
        OrderContext context = provider.getObject();
    }
}
```
ObjectFactory는 최소 DL 기능만 제공하지만, ObjectProvider는 확장된 DL 기능과 편의 기능을 제공합니다.

```java
provider.ifAvailable(...);
provider.stream();
provider.getIfUnique();
```

#### Provider#get()

Provider는 JSR-330 표준 인터페이스입니다.

```java
@Component
@RequiredArgsConstructor
public class OrderService {

    private final Provider<OrderContext> provider;

    public void order() {
        OrderContext context = provider.get();
    }
}
```

- 자바 표준 기반
- 스프링 의존성 감소
- 기능은 단순

즉, 스프링에 종속되지 않는 코드를 원할 때 사용할 수 있습니다.

## 1.3.2 스코프

### 스코프의 종류

스코프는 빈이 살아있는 범위를 의미합니다.

스프링은 다양한 스코프를 제공합니다.

- singleton : 애플리케이션 전체에서 하나
- prototype : 요청할 때마다 새 객체 생성
- request   : HTTP 요청마다 하나
- session   : 세션마다 하나
- application : ServletContext 범위

기본적으로는 singleton을 사용하지만, 웹 애플리케이션에서는 request나 session 스코프도 자주 사용됩니다.

### 스코프 빈의 사용 방법

예를 들어 사용자 요청별 데이터를 저장하는 빈을 만든다고 해봅시다.

```java
@Component
@Scope("request")
public class RequestData {

    private String userId;

    public void setUserId(String userId) {
        this.userId = userId;
    }
}
```

문제는 request 스코프 빈을 싱글톤 빈에 바로 주입하면 에러가 발생할 수 있다는 점입니다.

```java
@Service
@RequiredArgsConstructor
public class OrderService {

    private final RequestData requestData;
}
```

왜냐하면 싱글톤 빈 생성 시점에는 아직 HTTP 요청이 존재하지 않기 때문입니다.

이를 해결하기 위해 스프링은 프록시 기반 방식을 제공합니다.

```java
@Component
@Scope(
    value = "request",
    proxyMode = ScopedProxyMode.TARGET_CLASS
)
public class RequestData {
}
```

이렇게 하면 실제 객체 대신 프록시 객체가 주입됩니다.

- 싱글톤 빈 생성 시 → 실제 request 객체 대신 proxy 주입
- 실제 요청 시점 → 진짜 request 객체 생성

즉, 객체 생성을 지연시키는 방식입니다.

#### proxyMode 종류

1. TARGET_CLASS

```java
@Component
@Scope(
    value = "request",
    proxyMode = ScopedProxyMode.TARGET_CLASS
)
public class RequestData {
}
```

이 방식은 CGLIB 기반 프록시를 생성합니다.

즉, 실제 클래스를 상속한 프록시 객체 생성합니다.

따라서 인터페이스가 없어도 사용할 수 있습니다.

실무에서 가장 많이 사용하는 방식입니다.

2. INTERFACES

```java
@Component
@Scope(
    value = "request",
    proxyMode = ScopedProxyMode.INTERFACES
)
public class RequestDataImpl implements RequestData {
}
```

이 방식은 JDK Dynamic Proxy 기반입니다.

즉, 인터페이스 기반 프록시 생성합니다.

따라서 반드시 인터페이스가 필요합니다.

### 커스텀 스코프와 상태를 저장하는 빈 사용하기

스프링 빈은 기본적으로 싱글톤이기 때문에 상태를 저장하는 데 주의해야 합니다.

예를 들어 아래 코드는 위험합니다.

```java
@Service
public class OrderService {

    private int totalPrice;

    public void addPrice(int price) {
        totalPrice += price;
    }
}
```

싱글톤 빈은 여러 요청이 동시에 공유합니다.

따라서 여러 사용자가 동시에 접근하면 값이 섞일 수 있습니다.

```
A 사용자 요청 → totalPrice = 1000
B 사용자 요청 → totalPrice = 5000

상태 충돌 발생 가능
```

그래서 스프링에서는 가능한 한 상태를 저장하지 않는(stateless) 구조를 권장합니다.

상태가 꼭 필요하다면 다음과 같은 방법을 고려할 수 있습니다.

1. 메서드 파라미터 사용
2. 프로토타입 빈 사용
3. request/session 스코프 사용
4. ThreadLocal 사용
5. 커스텀 스코프 정의

커스텀 스코프는 특정 범위 동안만 유지되는 빈을 직접 정의하는 방식입니다.

예를 들어:

- 주문 처리 동안만 유지
- 배치 작업 동안만 유지
- 특정 사용자 흐름 동안만 유지

이처럼 스프링의 스코프 기능은 단순히 객체 생성 범위를 정하는 것을 넘어, 상태 관리 전략과도 깊게 연결됩니다.

## 정리

프로토타입과 스코프는 단순히 “객체를 여러 개 만든다” 수준의 개념이 아닙니다.

어떤 객체를 어디까지 공유할 것인지, 상태를 어떻게 안전하게 관리할 것인지와 연결되는 중요한 설계 요소입니다.

특히 프로토타입 빈은 싱글톤 빈과 함께 사용할 때 DI만으로는 원하는 동작을 보장할 수 없기 때문에 DL 전략이 필요합니다.

또한 request, session 같은 웹 스코프는 프록시를 통해 실제 요청 시점에 객체를 생성하는 방식으로 동작합니다.

결국 스프링의 스코프는 애플리케이션의 상태 관리와 객체 생명주기를 유연하게 제어하기 위한 핵심 기능이라고 볼 수 있습니다.
